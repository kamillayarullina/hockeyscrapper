# Отчет по лабораторной работе №4

## Часть 7: Автоматическое тестирование и проверки качества (QA)

В рамках этой части была выстроена комплексная система тестирования и контроля качества для приложения HockeyScrapper.

### 1. Документация по тестированию

Вся стратегия тестирования, архитектура, цели по покрытию и описание автоматизированных проверок были задокументированы в файле `docs/testing.md`. Этот документ служит единым источником информации о том, как обеспечивается качество проекта.

### 2. Реализация тестов

Были реализованы тесты нескольких типов:

*   **Unit-тесты**: Проверяют изолированные компоненты системы. Например, `tests/test_security.py` тестирует корректность хеширования и верификации паролей.
*   **Integration-тесты**: `tests/test_api_integration.py` проверяет взаимодействие компонентов через API, эмулируя действия пользователя (регистрация, логин, восстановление пароля).
*   **Автотесты качества (QRT)**: Были созданы три специальных теста для проверки нефункциональных требований:
    *   **QRT-01 (Отказоустойчивость)**: `tests/test_fault_tolerance.py` — проверяет, что парсер корректно обрабатывает сетевые ошибки (5xx) и выполняет повторные попытки, не прекращая работу аварийно.
    *   **QRT-02 (Конфиденциальность)**: `tests/test_auth_confidentiality.py` — гарантирует, что защищенные эндпоинты API возвращают ошибку 401 при попытке доступа без валидного JWT-токена.
    *   **QRT-03 (Производительность)**: `tests/test_notification_performance.py` — измеряет скорость работы движка уведомлений под нагрузкой, чтобы убедиться в соблюдении лимита производительности.

### 3. Дополнительная QA-проверка: Аудит безопасности зависимостей

В CI-пайплайн была добавлена автоматическая проверка всех сторонних библиотек из `requirements.txt` на наличие известных уязвимостей. Для этого используется инструмент **`pip-audit`**, который обращается к официальной базе данных уязвимостей PyPI.

**Цель**: Предотвратить атаки на цепочку поставок (supply-chain attacks), когда уязвимость в используемой библиотеке может скомпрометировать все приложение.

Если `pip-audit` находит уязвимость, сборка проекта немедленно прерывается, что блокирует развертывание потенциально опасного кода.

## Часть 8: Настройка CI (непрерывной интеграции)

Для автоматизации всех проверок был настроен CI-пайплайн с использованием GitHub Actions. Конфигурация находится в файле `.github/workflows/tests.yml`.

Пайплайн выполняет следующие шаги при каждом `push` или `pull request` в ветку `main`:

1.  **Checkout**: Загрузка последней версии кода из репозитория.
2.  **Set up Python**: Установка нужной версии Python (3.11).
3.  **Install dependencies**: Установка всех зависимостей из `requirements.txt`.
4.  **Install Playwright**: Установка браузеров, необходимых для работы парсера.
5.  **Run `pip-audit`**: Запуск аудита безопасности зависимостей.
6.  **Run `Bandit`**: Запуск статического анализатора для поиска уязвимостей в коде.
7.  **Run tests with coverage**: Запуск всех тестов (unit, integration, QRT) с помощью `pytest` и сбор отчета о покрытии кода.
8.  **Upload coverage report**: Выгрузка отчета о покрытии в виде артефакта сборки для последующего анализа.

Таким образом, обеспечен полный цикл автоматической проверки качества и безопасности кода при каждом изменении.
